# Lock 源码阅读

## 一、源代码

`Lock` 接口提供了比 `synchronized` 方法和代码块更丰富的加锁能力。它允许开发者用更灵活的方式组织加锁与解锁逻辑，不同实现还可能提供完全不同的语义，例如顺序保证、非可重入、死锁检测，以及多个关联的 `Condition` 对象。

从接口注释可以看出，`Lock` 的核心价值不只是“加锁”，而是把“锁的获取方式、是否可中断、是否可超时、如何配合条件变量、实现者需要遵守哪些内存语义”都显式定义出来。它尤其适用于需要跨作用域加锁解锁、需要尝试加锁、可中断加锁、定时等待加锁等更复杂的并发控制场景。

```java
package java.util.concurrent.locks;

public interface Lock {

    void lock();

    void lockInterruptibly() throws InterruptedException;

    boolean tryLock();

    boolean tryLock(long time, TimeUnit unit) throws InterruptedException;

    void unlock();

    Condition newCondition();
}
```

## 二、接口职责

### 1、定义显式锁能力

`Lock` 定义了一套显式锁 API，用于控制多个线程对共享资源的访问。与 `synchronized` 相比，它把加锁、解锁、中断、超时等行为暴露为独立方法，使调用者可以更灵活地安排控制流程。

### 2、弥补块结构锁的局限

`synchronized` 的好处是语法简单，并且锁的获取与释放天然跟随代码块作用域；但它也带来了限制，例如多把锁必须按嵌套作用域释放。`Lock` 的设计正是为了打破这种局限，让锁可以在不同作用域获取和释放。

### 3、为高级并发语义留出扩展点

接口注释强调，不同 `Lock` 实现可以提供不同特性，例如：

- 顺序保证
- 非可重入语义
- 死锁检测
- 多个 `Condition` 对象

这说明 `Lock` 既是调用方 API，也是实现方向外承诺并发语义的契约。

## 三、方法

### 1、`lock()`

该方法用于获取锁。

如果锁当前不可用，调用线程会进入等待，直到成功获得锁。接口注释还特别说明，实现类可以在检测到错误使用方式时抛出运行时异常，例如可能导致死锁的调用。

这个方法的特点是：

- 会阻塞等待
- 不提供超时能力
- 不以抛出 `InterruptedException` 的方式响应中断

### 2、`lockInterruptibly()`

该方法表示“可中断地获取锁”。

如果锁可用，会立即返回；如果锁不可用，线程会等待，直到发生以下两种情况之一：

- 当前线程成功获取锁
- 当前线程在等待期间被中断，并且实现支持中断获取锁

如果线程在进入方法前已带有中断标记，或在等待期间被中断，则会抛出 `InterruptedException`，并清除当前线程的中断状态。

这个方法适合用于需要支持取消、超时控制链路、任务编排等场景。

### 3、`tryLock()`

该方法用于“尝试获取锁但不阻塞等待”。

如果调用瞬间锁可用，则立即获取并返回 `true`；否则立刻返回 `false`。它适合用于不希望线程挂起，而是希望在拿不到锁时执行降级逻辑、跳过处理或稍后重试的场景。

### 4、`tryLock(long time, TimeUnit unit)`

该方法表示“在指定时间内尝试获取锁”。

它融合了两个维度的语义：

- 最长等待时间
- 可中断等待

调用后可能出现三类结果：

- 成功拿到锁，返回 `true`
- 超过等待时间仍未拿到锁，返回 `false`
- 等待期间被中断，抛出 `InterruptedException`

如果传入时间小于等于 `0`，则不会真正等待。

### 5、`unlock()`

该方法用于释放锁。

接口注释指出，实现类通常会限制只有持有锁的线程才能执行释放。如果违反限制，可能抛出运行时异常。因此调用方必须确保释放行为与获取行为严格配对。

### 6、`newCondition()`

该方法返回一个绑定到当前 `Lock` 实例的 `Condition` 对象。

在等待条件前，调用线程必须先持有该锁。调用 `Condition.await()` 时，会先原子性释放锁，等待结束后再重新获取锁。这让 `Lock` 可以像传统监视器的 `wait/notify` 一样支持条件等待，但能力更显式、可扩展性更强。

## 四、实现与使用约束

### 1、推荐使用方式

接口注释明确给出了推荐模式：获取锁后，务必把受保护代码放进 `try-finally` 中，在 `finally` 里执行 `unlock()`，以确保异常情况下也能释放锁。

### 2、作用域更灵活但责任更大

`Lock` 允许在不同作用域获取和释放锁，这比 `synchronized` 更灵活，但也意味着开发者必须自己保证：

- 每次成功加锁后最终都会释放
- 涉及共享资源的代码都被锁保护
- 多把锁的获取与释放顺序不会导致死锁

### 3、内存语义要求

接口注释规定，所有 `Lock` 实现都必须提供与内置监视器锁相同的内存同步语义：

- 成功的 `lock` 具有与成功加锁动作相同的内存同步效果
- 成功的 `unlock` 具有与成功解锁动作相同的内存同步效果

但失败的加锁、失败的解锁，以及可重入场景下的重复加解锁，不要求产生同样的内存同步效果。

### 4、中断与超时语义因实现而异

接口也特别提醒，实现类不一定能以完全相同的性能和语义支持三种获取方式：

- 不可中断获取
- 可中断获取
- 带超时获取

因此真正使用某个具体实现时，还必须阅读该实现类自己的文档说明。

### 5、不要混淆对象监视器与显式锁

`Lock` 实例本身也是普通对象，因此理论上仍可以作为 `synchronized` 的目标对象，但接口文档明确指出，这种做法与调用 `lock()` 系列方法之间没有规定好的语义关系，容易造成误解，通常不建议这样使用。
