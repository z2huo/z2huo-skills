# Java 源码阅读与文档生成

## 一、Skill 目标

本 Skill 用于阅读 Java 源码，并基于源码内容生成结构化的中文 Markdown 文档。

重点覆盖以下三类典型场景：

- 普通类：同时包含属性、构造方法、业务方法或工具方法
- Pojo 类：以属性、Getter、Setter 为主，通常没有明显业务逻辑
- 接口：以方法声明和接口语义说明为主，不包含实例属性

示例源码类型：

- 普通类参考：`references/ArrayList.java`
- Pojo 类参考：`references/ConsumerProperties.java`
- 接口参考：`references/Lock.java`

## 二、自动触发规则

- 当用户要求“阅读 Java 源码”时启用
- 当用户要求“生成 Java 源码解读文档”时启用
- 当用户要求“把 Java 类/接口整理成 Markdown”时启用
- 当用户提供 `.java` 文件并希望输出中文说明、源码分析、字段说明、方法说明时启用
- 当任务核心是“翻译 Java 注释并整理为文档”，而不是直接修改 Java 源码时启用

## 三、输入识别规则

开始输出前，先判断当前 Java 源码属于哪一类：

### 1、普通类

满足任一特征即可判断为普通类：

- 存在属性
- 存在一个或多个构造方法
- 存在除 Getter/Setter 之外的业务方法、工具方法或核心算法方法

例如：

- `ArrayList`

### 2、Pojo 类

满足以下特征时优先判断为 Pojo 类：

- 主要由属性组成
- 方法基本为 Getter、Setter、`isXxx()`、少量框架辅助方法
- 几乎不包含业务逻辑

例如：

- `ConsumerProperties`

### 3、接口

满足以下特征时判断为接口：

- 使用 `interface` 定义
- 主要内容为方法声明、常量、默认方法或静态方法
- 文档重心是接口职责与方法契约

例如：

- `Lock`

## 四、输出总原则

生成 Markdown 文档时，必须遵守以下约束：

1. 全文使用中文输出。
2. 严格基于源码可见信息进行总结，不要编造不存在的实现细节。
3. 原始英文注释必须翻译为中文，再写入文档。
4. 文档结构应根据“普通类 / Pojo 类 / 接口”自动裁剪，不要强行保留无意义章节。
5. 如果某个章节不存在实际内容，可以省略该章节，不要保留空标题。
6. 说明内容先写“职责和作用”，再写“关键成员/方法”，最后写“使用注意点”。

## 五、源代码章节规则

无论是普通类、Pojo 类还是接口，都必须先输出“源代码”章节。

### 1、源代码章节固定要求

源代码章节使用固定结构，要求如下：

1. 先写类或接口顶部注释的中文翻译与摘要。
2. 再粘贴清理后的 Java 源码代码块。
3. 代码块中必须移除所有注释，包括：
   - 类注释
   - 属性注释
   - 方法注释
   - 行注释
   - 块注释
   - `import`
4. 清理后的源码必须保留这些结构：
   - `package`
   - 类 / 接口声明
   - 属性
   - 构造方法
   - 方法签名或方法实现
5. 不要为了精简而删掉字段、方法、构造方法等核心结构。
6. 对于以标准 Getter、Setter 为主的 Pojo 类，“源代码”章节默认不展示标准 Getter、Setter，只保留字段，以及像框架回填、校验、转换这类非标准访问器方法。

### 2、源代码章节推荐格式

````markdown
## 一、源代码

这里先写类或接口顶部注释的中文翻译和摘要。

```java
package xxx;

public class Demo {

    private String name;

    public Demo() {
    }

    public String getName() {
        return name;
    }
}
```
````

## 六、普通类文档结构

当源码被识别为“普通类”时，优先使用以下结构：

```markdown
## 一、源代码

## 二、类职责

## 三、属性

## 四、构造方法

## 五、方法

## 六、使用要点
```

### 1、类职责

需要概括：

- 这个类解决什么问题
- 在整体设计中承担什么角色
- 是否有线程安全、性能、容量、扩容、并发等关键特性

### 2、属性

属性章节需要：

- 先给出属性总览
- 再按字段逐个说明
- 说明字段的用途，而不是简单翻译字段名
- 如果是常量，说明它的默认值或设计目的

### 3、构造方法

构造方法章节需要：

- 先给出构造方法列表和差异
- 再分别说明每个构造方法的用途
- 指出入参含义、默认行为、异常条件

### 4、方法

方法章节需要：

- 优先保留核心方法
- 如果方法很多，允许按“容量管理”“读操作”“写操作”“遍历操作”等分组说明
- 对每个方法写清楚：作用、关键参数、返回值、异常、关键实现思路
- 如果方法是明显的内部辅助方法，要说明它服务于哪个主流程

### 5、使用要点

优先总结：

- 线程安全要求
- 性能特征
- 使用限制
- 容易误用的地方

## 七、Pojo 类文档结构

当源码被识别为“Pojo 类”时，优先使用以下结构：

```markdown
## 一、源代码

## 二、对象职责

## 三、属性

## 四、访问方法

## 五、使用要点
```

### 1、对象职责

这里重点说明：

- 这个对象承载什么配置或数据
- 它通常在哪类场景中被使用
- 它是业务对象、配置对象、传输对象还是状态对象

### 2、属性

属性章节是 Pojo 文档的重点，要求：

- 按属性逐个解释
- 优先翻译字段注释
- 明确默认值、可选值、特殊取值含义
- 如果字段之间存在覆盖关系、联动关系、互斥关系，必须指出

### 3、访问方法

对于 Getter/Setter 为主的 Pojo：

- 不需要逐个机械解释所有 Getter/Setter
- “源代码”章节默认不展示标准 Getter/Setter
- 应概括为“用于读取和设置对应属性”
- 如果存在特殊方法，例如 `populateBindingName` 这类框架回填方法，需要单独说明
- 如果某些方法并非标准访问器，应单独列出其用途

### 4、使用要点

优先总结：

- 默认值影响
- 配置优先级或覆盖关系
- 仅供框架使用的方法
- 使用者需要避免的误配置

## 八、接口文档结构

当源码被识别为“接口”时，优先使用以下结构：

```markdown
## 一、源代码

## 二、接口职责

## 三、方法

## 四、实现与使用约束
```

### 1、接口职责

这里需要说明：

- 接口定义了什么能力
- 它与语言内置机制或其他接口相比多了什么
- 典型使用场景是什么

### 2、方法

方法章节需要：

- 逐个解释方法契约
- 说明调用行为、阻塞语义、中断语义、返回结果、异常语义
- 如果注释中存在“实现注意事项”或“实现约束”，要翻译并写入说明
- 如果接口方法之间有配套关系，例如加锁和解锁，需要指出调用习惯

### 3、实现与使用约束

优先总结：

- 实现类需要遵守的语义
- 调用方需要遵守的使用方式
- 中断、超时、线程安全、资源释放等注意点

## 九、翻译规则

翻译 Java 注释时，遵循以下要求：

1. 以“准确表达原意”为第一优先级，不要只做逐词直译。
2. 遇到 Java 专有术语时，保留常见技术表达。
3. `Lock`、`Condition`、`List`、`Message` 等类型名默认保留原名，必要时在中文中解释其含义。
4. `Default`、`NOTE`、`since`、`throws`、`return`、`param` 等说明要翻译为自然中文语句。
5. 对于超长类注释，先给出整体摘要，再分点翻译关键约束和设计意图。

## 十、内容裁剪规则

为避免文档机械和冗余，输出时遵循以下规则：

1. 如果类没有属性，则不要生成“属性”章节。
2. 如果类没有显式构造方法，则不要生成“构造方法”章节。
3. 如果 Pojo 只有标准 Getter/Setter，不逐个罗列每个访问器，可合并说明；在“源代码”章节中也不要展示这些标准 Getter/Setter。
4. 如果接口方法很多，可以先给方法总览，再按方法逐个解释。
5. 如果源码片段被截断，应明确说明“当前分析基于提供的片段，不代表完整源码”。

## 十一、推荐执行步骤

每次执行本 Skill 时，按照下面顺序处理：

1. 阅读 Java 顶部注释，判断整体职责。
2. 识别源码类型：普通类、Pojo 类或接口。
3. 提取类名、包名、导入信息、属性、构造方法、方法；其中导入信息仅用于理解依赖关系，不写入“源代码”章节。
4. 翻译类注释，并整理为“源代码”章节的导语。
5. 生成去注释后的 Java 代码块。
6. 根据源码类型选择对应的 Markdown 结构。
7. 对属性、构造方法、方法进行中文说明。
8. 最后总结使用要点、约束、风险或注意事项。

## 十二、输出要求

每次最终输出的 Markdown 文档，必须满足以下要求：

1. 标题层级清晰，章节顺序稳定。
2. 所有段落之间保留空行。
3. 不使用空洞描述，例如“这是一个很重要的方法”。
4. 不重复照抄源码签名作为解释，应补充语义说明。
5. 如果是 Pojo，重点写属性含义；如果是接口，重点写方法契约；如果是普通类，重点写结构与核心方法。
