# Java 源码阅读与文档生成

## 一、Skill 目标

本 Skill 用于阅读 Java 源码，并基于源码内容生成结构化的中文 Markdown 文档。

常见场景大致包括以下几类，但这些只是示例，不是强制分类前提：

- 普通类：同时包含属性、构造方法、业务方法或工具方法
- Pojo 类：以属性、Getter、Setter 为主，通常没有明显业务逻辑
- 接口：以方法声明和接口语义说明为主，不包含实例属性

## 二、自动触发规则

- 当用户要求“阅读 Java 源码”时启用
- 当用户要求“生成 Java 源码解读文档”时启用
- 当用户要求“把 Java 类/接口整理成 Markdown”时启用
- 当用户提供 Java 源码并希望输出中文说明、源码分析、字段说明、方法说明时启用
- 当任务核心是“翻译 Java 注释并整理为文档”，而不是直接修改 Java 源码时启用

## 三、输入与结构识别规则

### 1、输入来源

本 Skill 支持以下几种 Java 源码输入来源：

- 本地文件路径形式的 `.java` 文件
- 用户上传的 `.java` 附件
- 用户直接在对话框中粘贴的 Java 代码或代码片段

只要输入内容本质上是 Java 源码，就可以按本 Skill 的规则进行识别和输出，不要求必须来自本地文件。

### 2、识别原则

- 不要求先把源码强制归类为普通类、Pojo 类或接口
- 普通类、Pojo 类、接口只是常见场景示例，不是互斥模板
- 应先识别源码中实际存在的结构元素，再决定 Markdown 章节如何组织
- 同一份 Java 源码中，只要存在某类结构，就遵循对应的结构规则

### 3、结构化命令输入格式

除自然语言输入外，也支持以下约定格式：

```text
/java-source-code-reading input_file output_file
```

按位置解析规则如下：

- `/java-source-code-reading` 后面的第一个文件参数视为输入文件
- `/java-source-code-reading` 后面的第二个文件参数视为输出文件
- 必须校验第一个文件是否以 `.java` 结尾
- 必须校验第二个文件是否以 `.md` 结尾
- 如果第一个文件不是 `.java`，则本次指定的输入文件不生效
- 如果第二个文件不是 `.md`，则本次指定的输出文件不生效
- 如果输入文件和输出文件任一校验失败，则该命令中的对应文件参数按“未指定”处理
- 当结构化命令中的文件参数失效后，仍可继续按自然语言内容、附件或粘贴代码规则处理

### 4、输出位置规则

输出位置按输入来源决定：

- 如果输入是一个明确的本地 `.java` 文件路径，则默认在该 `.java` 文件同目录下创建同名 `.md` 文件
- 如果用户明确指定了输出文件路径或输出目录，则优先遵循用户指定
- 如果输入是用户上传的 `.java` 附件，或用户直接在对话框中粘贴的 Java 代码，则不能默认落盘
- 对于附件或粘贴代码，必须先询问用户是否需要输出到具体文件中
- 如果用户指定了具体输出文件，则写入对应文件
- 如果用户没有指定输出文件，或明确表示不需要落盘，则直接以文本形式输出 Markdown 内容，不创建文件

对结构化命令输入，还需要补充以下约束：

- 当 `/java-source-code-reading input_file output_file` 中的第一个参数是合法 `.java` 文件时，按显式输入文件处理
- 当第二个参数是合法 `.md` 文件时，按显式输出文件处理
- 当第二个参数缺失时，如果第一个参数是本地 `.java` 文件，则仍可按“同目录同名 `.md`”默认规则输出
- 当两个参数都不合法时，这种结构化命令写法本身不报错，但文件参数不生效，后续回退到普通输入识别规则

### 5、常见场景示例

#### 5.1 普通类

满足任一特征即可判断为普通类：

- 存在属性
- 存在一个或多个构造方法
- 存在除 Getter/Setter 之外的业务方法、工具方法或核心算法方法

例如：

- `ArrayList`

#### 5.2 Pojo 类

满足以下特征时优先判断为 Pojo 类：

- 主要由属性组成
- 方法基本为 Getter、Setter、`isXxx()`、少量框架辅助方法
- 几乎不包含业务逻辑

例如：

- `ConsumerProperties`

#### 5.3 接口

满足以下特征时判断为接口：

- 使用 `interface` 定义
- 主要内容为方法声明、常量、默认方法或静态方法
- 文档重心是接口职责与方法契约

例如：

- `Lock`

### 6、结构元素识别

开始输出前，应识别源码中实际存在的结构元素，包括但不限于：

- 属性：成员变量、常量、配置字段、状态字段
- 构造方法：无参构造、全参构造、重载构造
- 标准访问器：`getXxx()`、`setXxx()`、`isXxx()` 这类常规访问方法
- 非标准方法：业务方法、工具方法、校验方法、转换方法、框架回填方法
- 接口方法：接口中的抽象方法、默认方法、静态方法

## 四、输出总原则

生成 Markdown 文档时，必须遵守以下约束：

1. 全文使用中文输出。
2. 严格基于源码可见信息进行总结，不要编造不存在的实现细节。
3. 原始英文注释必须翻译为中文，再写入文档。
4. 文档结构应根据源码中实际存在的结构元素自动裁剪，不要强行保留无意义章节。
5. 如果某个章节不存在实际内容，可以省略该章节，不要保留空标题。
6. 说明内容先写“职责和作用”，再写“关键成员/方法”，最后写“使用注意点”。
7. 生成 Markdown 时，格式必须遵循全局 `markdown-generate-format` Skill 的规范。
8. 如果当前项目中存在并命中 `obsidian-markdown` 这个项目级 Skill 的生效场景，则还必须在全局 Markdown 规范基础上，同时遵循 `obsidian-markdown` Skill 的规范。

## 五、源代码章节规则

无论是哪种 Java 源码场景，都必须先输出“源代码”章节。

### 1、源代码章节固定要求

源代码章节使用固定结构，要求如下：

1. 先写类或接口顶部注释的中文翻译，不额外添加摘要或总结。
2. 再粘贴清理后的 Java 源码代码块。
3. 代码块中必须移除所有注释，包括：
   - 类注释
   - 属性注释
   - 方法注释
   - 行注释
   - 块注释
   - `import`
4. 清理后的源码必须保留这些结构：
   - `package`
   - 类 / 接口声明
   - 属性
   - 构造方法
   - 方法签名或方法实现
5. 不要为了精简而删掉字段、方法、构造方法等核心结构。
6. 对于源码中的标准 Getter、Setter、`isXxx()` 访问器，“源代码”章节默认不展示这些标准访问器，只保留字段，以及像框架回填、校验、转换这类非标准方法。
7. 如果属性或方法带有 `@Deprecated` 注解，则视为已废弃成员；在“源代码”章节中不再展示这些已废弃的属性或方法。

### 2、源代码章节推荐格式

````markdown
## 一、源代码

这里先写类或接口顶部注释的中文翻译。

```java
package xxx;

public class Demo {

    private String name;

    public Demo() {
    }
}
```
````

## 六、章节组织规则

文档章节应按“源码里实际存在什么结构，就输出什么章节”的原则组织，而不是先把源码固定归类为普通类、Pojo 类或接口，再套某一套独占模板。

### 1、基础结构

所有文档都以以下结构为起点：

```markdown
## 一、源代码

## 二、职责说明
```

其中：

- `源代码` 章节必选
- `职责说明` 章节建议保留，但标题可以根据语义写成“类职责”“对象职责”“接口职责”

在此基础上，再根据源码中实际存在的结构元素，按需追加下面的章节：

- `属性`
- `构造方法`
- `方法`
- `使用要点`
- `实现与使用约束`

### 2、属性章节

只要源码中存在字段、常量、配置项或状态成员，就应输出“属性”章节。

属性章节需要：

- 先给出属性总览
- 再按字段逐个说明
- 忽略 `serialVersionUID` 这类仅用于 Java 序列化版本控制的字段，不在属性章节中展开说明
- 如果字段带有 `@Deprecated` 注解，则视为已废弃字段，不在属性章节中展开说明
- 说明字段用途，而不是简单翻译字段名
- 如果是常量，说明默认值、设计目的或语义边界
- 如果字段之间存在覆盖关系、联动关系、互斥关系，必须指出

### 3、构造方法章节

只要源码中存在显式构造方法，就应输出“构造方法”章节。

构造方法章节需要：

- 先给出构造方法列表和差异
- 再分别说明每个构造方法的用途
- 指出入参含义、默认行为、异常条件

### 4、方法章节

只要源码中存在非标准方法、业务方法、工具方法、接口方法、默认方法或静态方法，就应输出“方法”章节。

方法章节需要：

- 优先保留核心方法
- 如果方法很多，允许按职责分组说明
- 如果方法带有 `@Deprecated` 注解，则视为已废弃方法，不在“方法”章节中展开说明
- 对每个方法写清楚：作用、关键参数、返回值、异常、关键语义或实现思路
- 如果是接口方法，重点写方法契约、阻塞语义、中断语义、超时语义、返回结果和异常语义
- 如果源码中同时存在标准 Getter、Setter、`isXxx()` 访问器，不需要为它们单独设置“访问方法”章节，通常一句话概括为“用于读取和设置对应属性”即可
- 如果存在特殊方法，例如 `populateBindingName` 这类框架回填方法，需要在“方法”章节中单独说明
- 如果方法是明显的内部辅助方法，要说明它服务于哪个主流程

### 5、职责说明章节

职责说明章节的标题不是固定的，可以根据源码语义使用下列表达之一：

- `类职责`
- `对象职责`
- `接口职责`
- `职责说明`

这里需要说明：

- 这个类、对象或接口解决什么问题
- 在整体设计中承担什么角色
- 是否有线程安全、性能、配置优先级、实现约束等关键特性

### 6、使用要点与约束章节

当源码中存在使用限制、线程安全要求、实现约束、配置优先级、互斥关系等内容时，应补充相应章节。

可选标题包括：

- `使用要点`
- `实现与使用约束`

优先说明：

- 线程安全要求
- 性能特征
- 配置优先级或覆盖关系
- 仅供框架使用的方法
- 中断、超时、资源释放、实现方约束
- 使用者容易误用的地方

## 九、翻译规则

翻译 Java 注释时，遵循以下要求：

1. 以“准确表达原意”为第一优先级，不要只做逐词直译。
2. 遇到 Java 专有术语时，保留常见技术表达。
3. `Lock`、`Condition`、`List`、`Message` 等类型名默认保留原名，必要时在中文中解释其含义。
4. `Default`、`NOTE`、`since`、`throws`、`return`、`param` 等说明要翻译为自然中文语句。
5. 对于超长类注释，按原有语义和段落结构翻译为中文，必要时可分段呈现，但不要额外补充摘要。

## 十、内容裁剪规则

为避免文档机械和冗余，输出时遵循以下规则：

1. 如果源码没有属性，则不要生成“属性”章节。
2. 如果源码没有显式构造方法，则不要生成“构造方法”章节。
3. 如果源码中存在标准 Getter/Setter，不逐个罗列每个访问器，可合并说明；在“源代码”章节中也不要展示这些标准 Getter/Setter。
4. 如果存在 `serialVersionUID` 字段，在属性相关章节中忽略该字段，不单独说明。
5. 如果字段或方法带有 `@Deprecated` 注解，则在“源代码”章节以及后续属性、方法相关章节中都忽略，不单独说明。
6. 如果方法很多，可以先给方法总览，再按方法逐个解释。
7. 如果源码片段被截断，应明确说明“当前分析基于提供的片段，不代表完整源码”。

## 十一、推荐执行步骤

每次执行本 Skill 时，按照下面顺序处理：

1. 阅读 Java 顶部注释，判断整体职责。
2. 识别源码中实际存在的结构元素：属性、构造方法、标准 Getter/Setter、非标准方法、接口方法。
3. 优先检查用户是否使用了 `/java-source-code-reading input_file output_file` 这种结构化命令格式。
4. 如果使用了结构化命令格式，则校验第一个参数是否为 `.java`，第二个参数是否为 `.md`；校验失败的参数按未指定处理。
5. 判断输入来源，并确定输出方式：本地 `.java` 文件默认同目录生成同名 `.md`；附件或粘贴代码先询问是否需要落盘。
6. 提取类名、包名、导入信息、属性、构造方法、方法；其中导入信息仅用于理解依赖关系，不写入“源代码”章节。
7. 如果字段中存在 `serialVersionUID`，则在后续属性章节中忽略该字段。
8. 后续生成 Markdown 时，默认始终遵循全局 `markdown-generate-format` Skill 的规则。
9. 判断当前输出是否命中 `obsidian-markdown` 这个项目级 Skill 的生效场景；如果命中，则在遵循全局 Markdown 规范的基础上，再同时遵循该 Skill 的规则。
10. 如果属性或方法带有 `@Deprecated` 注解，则在“源代码”章节以及后续属性、方法相关章节中统一忽略。
11. 翻译类注释，并整理为“源代码”章节的导语。
12. 生成去注释后的 Java 代码块。
13. 根据实际存在的结构元素组织 Markdown 章节，而不是先强制按普通类、Pojo 类或接口分类。
14. 对属性、构造方法、业务方法、接口方法和特殊方法分别进行中文说明；标准 Getter、Setter、`isXxx()` 访问器不单独成章。
15. 最后补充使用要点、实现约束、风险或注意事项。

## 十二、输出要求

每次最终输出的 Markdown 文档，必须满足以下要求：

1. 标题层级清晰，章节顺序稳定。
2. 所有段落之间保留空行。
3. 不使用空洞描述，例如“这是一个很重要的方法”。
4. 不重复照抄源码签名作为解释，应补充语义说明。
5. 如果存在属性，重点写属性含义；如果存在标准 Getter/Setter，重点写访问方式；如果存在业务方法或接口方法，重点写方法契约、行为与约束。
6. Markdown 标题编号、段落换行和文案风格默认遵循全局 `markdown-generate-format` 的要求；如果当前项目中命中 `obsidian-markdown` 的生效场景，则还要同时遵循该项目级 Skill 的要求。
7. 如果输入来自本地 `.java` 文件，默认输出到同目录同名 `.md` 文件。
8. 如果输入来自附件或对话框粘贴代码，只有在用户明确指定输出文件时才落盘；否则直接文本输出即可。
9. 如果用户使用 `/java-source-code-reading input_file output_file` 这种结构化命令格式，则必须先校验文件后缀；只有合法 `.java` 输入和合法 `.md` 输出才生效。
